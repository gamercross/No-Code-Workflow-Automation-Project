## ⚙️ Chapter 2. 자동화 개념 이해

# 📖 자동화란 무엇인가?

자동화(Automation)는 사람이 반복적으로 수행하는 작업을 시스템이 대신 수행하도록 만드는 기술을 의미한다. 과거에는 이러한 자동화를 구현하기 위해 프로그래밍 언어를 사용하여 직접 코드를 작성해야 했지만, 최근에는 **노코드(No-Code)** 및 **로우코드(Low-Code)** 플랫폼의 발전으로 프로그래밍 경험이 없어도 자동화 시스템을 구축할 수 있게 되었다.

대표적인 노코드 자동화 플랫폼인 **Make**와 **Zapier**는 다양한 서비스를 서로 연결하여 하나의 업무 흐름(Workflow)을 구성할 수 있도록 지원한다. 사용자는 각 서비스를 시각적으로 연결하는 것만으로도 복잡한 자동화 프로세스를 구현할 수 있으며, 반복 업무를 줄이고 업무 효율을 크게 향상시킬 수 있다.

자동화 워크플로우는 일반적으로 **Trigger → 조건 확인 → Action**의 순서로 동작하며, 필요에 따라 여러 개의 조건 분기와 작업을 추가하여 더욱 복잡한 프로세스를 구성할 수 있다.

---

# 🎯 Trigger (시작 이벤트)

## Trigger란?

**Trigger**는 자동화를 시작시키는 이벤트(Event)를 의미한다.

즉, 특정 조건이나 이벤트가 발생하면 자동화 시스템은 이를 감지하고 미리 정의된 워크플로우를 실행하게 된다.

Trigger는 모든 자동화의 시작점이며, Trigger가 발생하지 않으면 이후의 작업(Action)도 실행되지 않는다.

### 💡 대표적인 Trigger 예시

* 📧 Gmail에 새로운 이메일이 도착했을 때
* 📝 Google Form에 새로운 응답이 제출되었을 때
* 📂 Google Drive에 새로운 파일이 업로드되었을 때
* 📄 Google Docs 문서가 생성되었을 때
* 🌐 Webhook으로 외부 요청이 들어왔을 때
* 💬 Slack 또는 Discord에 새로운 메시지가 작성되었을 때

---

## Trigger 동작 과정

```text
이벤트 발생

↓

Trigger 감지

↓

Workflow 실행 시작
```

Trigger는 자동화의 시작 버튼과 같은 역할을 수행하며, 이후 연결된 Action들이 순차적으로 실행된다.

---

# ⚡ Action (실행 작업)

## Action이란?

**Action**은 Trigger가 발생한 이후 실제로 수행되는 작업을 의미한다.

하나의 워크플로우에는 하나 이상의 Action이 포함될 수 있으며, 여러 Action을 순차적으로 연결하여 복잡한 업무를 자동화할 수 있다.

예를 들어 새로운 설문 응답이 제출되었다면,

* Google Sheets에 데이터 저장
* Slack으로 알림 전송
* 이메일 발송

과 같은 여러 작업을 연속적으로 수행할 수 있다.

---

## 💡 대표적인 Action 예시

* 📊 Google Sheets에 새로운 행 추가
* 💬 Slack 채널에 메시지 전송
* 📧 이메일 자동 발송
* 📁 Google Drive에 파일 저장
* 🤖 ChatGPT를 이용한 텍스트 요약
* 📱 Discord 알림 전송

---

## Action 동작 과정

```text
Trigger 발생

↓

Action 1

↓

Action 2

↓

Action 3

↓

자동화 완료
```

Action은 자동화 시스템의 핵심 기능으로, 실제 업무를 대신 수행하는 역할을 한다.

---

# 🔀 Filter (조건 분기)

## Filter란?

**Filter**는 특정 조건을 만족하는 경우에만 다음 단계를 실행하도록 하는 기능이다.

조건을 만족하지 못하면 해당 경로는 실행되지 않고 워크플로우가 종료되거나 다른 경로로 이동한다.

이를 통해 불필요한 작업을 줄이고 필요한 경우에만 자동화를 수행할 수 있다.

---

## Filter 예시

학생들의 시험 점수를 자동으로 처리한다고 가정하면 다음과 같이 구성할 수 있다.

```text
Google Form 제출

↓

점수 확인

↓

점수 ≥ 80점

↓

Slack 합격 알림
```

80점 미만이라면 Slack 알림은 실행되지 않는다.

---

## Filter의 장점

* ✅ 불필요한 작업 실행 방지
* ✅ 조건에 맞는 데이터만 처리
* ✅ 자동화 효율 향상
* ✅ 리소스 절약

---

# 🔀 Router (다중 분기)

## Router란?

**Router**는 하나의 워크플로우를 여러 개의 경로로 나누어 각각 다른 작업을 수행하도록 하는 기능이다.

Make에서는 Router를 사용하여 하나의 Trigger에서 여러 개의 작업 흐름을 동시에 설계할 수 있다.

---

## Router 예시

```text
Google Form 제출

↓

Router

├── 점수 ≥ 80
│      ↓
│   합격 시트 저장
│      ↓
│   Slack 합격 알림
│
└── 점수 < 80
       ↓
   불합격 시트 저장
       ↓
   Slack 재도전 안내
```

이처럼 하나의 Trigger에서도 조건에 따라 서로 다른 자동화 프로세스를 실행할 수 있다.

---

## Router의 장점

* 🔹 복잡한 업무를 하나의 워크플로우에서 처리 가능
* 🔹 여러 조건을 동시에 관리 가능
* 🔹 유지보수가 용이
* 🔹 시각적으로 이해하기 쉬운 구조 제공

---

# 🔄 Trigger · Filter · Router · Action의 관계

자동화 시스템은 일반적으로 다음과 같은 흐름으로 동작한다.

```text
Trigger

↓

Filter 또는 Router

↓

Action

↓

Action

↓

Workflow 종료
```

각 요소는 서로 독립적인 기능이 아니라 하나의 워크플로우 안에서 유기적으로 연결되어 자동화를 완성한다.

# ⚡ Zapier 구현

## 📌 구현 방식

Zapier는 **리스트(List) 기반 인터페이스**를 제공하며, 단계별로 Trigger와 Action을 추가하는 방식으로 워크플로우를 구성한다.

# 🚀 Chapter 5. 프로젝트 2 - 자유 주제 자동화 설계 및 구현

## 📖 프로젝트 개요

앞선 프로젝트에서는 동일한 자동화 워크플로우를 **Make**와 **Zapier**에서 각각 구현하며 플랫폼의 특징과 차이점을 비교하였다.

이번 프로젝트에서는 단순히 예제를 구현하는 것을 넘어, 실제 업무에서 반복적으로 발생할 수 있는 작업을 직접 선정하고 이를 자동화하는 것을 목표로 하였다.

자동화 대상은 실생활이나 업무 환경에서 자주 수행되는 작업일수록 자동화의 효과가 크다. 따라서 "얼마나 복잡한 기능을 구현했는가"보다 **반복되는 업무를 얼마나 효율적으로 개선할 수 있는가**에 중점을 두어 자동화를 설계하였다.

# 🎯 자동화 대상 업무 정의

이번 프로젝트에서 선정한 반복 업무는 다음과 같다.

## 📝 회의록 자동 요약 및 팀 채널 공유

회의가 끝난 후에는 일반적으로 다음과 같은 작업을 수행한다.

1. 회의록 작성
2. 주요 내용 정리
3. 결정 사항 요약
4. 팀원들에게 공유
5. 회의 이력 저장

이 과정은 매번 반복되며, 회의 시간이 길어질수록 요약과 공유에 많은 시간이 소요된다. 또한 사람이 직접 요약하는 과정에서 중요한 내용이 누락되거나 표현이 일관되지 않을 가능성도 있다.

이러한 문제를 해결하기 위해 **회의록 작성부터 요약, 공유, 기록까지의 과정을 자동화**하는 워크플로우를 설계하였다.

# ⚙️ 자동화 도구 선정

이번 프로젝트에서는 **Make**를 자동화 플랫폼으로 선정하였다.

## 📌 선정 이유

Make를 선택한 이유는 다음과 같다.

* ✅ 시각적인 워크플로우를 구성하기 쉽다.
* ✅ Router를 활용한 조건 분기가 편리하다.
* ✅ Google Workspace와의 연동이 우수하다.
* ✅ 제미나이 API 등 AI 서비스와의 연동이 용이하다.
* ✅ 무료 플랜에서도 프로젝트 구현이 가능하다.

특히 이번 프로젝트에서는 여러 서비스를 하나의 흐름으로 연결해야 했기 때문에, 전체 구조를 한눈에 확인할 수 있는 Make의 인터페이스가 적합하다고 판단하였다.

# 🏗️ 워크플로우 설계

전체 자동화 흐름은 다음과 같이 설계하였다.

```text

        │
        ▼
Trigger
        │
        ▼
회의 내용 가져오기
```

# 🗺️ 전체 워크플로우

이번 프로젝트의 자동화는 아래와 같은 흐름으로 동작한다.

```text
Google Docs 작성
        │
        ▼
🚀 Trigger
        │
        ▼
📄 문서 내용 추출
        │
        ▼
🤖 ChatGPT 요약
        │
        ▼
🔀 Filter
        │
 ┌──────┴──────┐
 │             │
 ▼             ▼
Slack 전송     종료
 │
 ▼
Google Sheets 저장
 │
 ▼
자동화 완료
```

각 단계는 이전 단계에서 생성된 데이터를 입력으로 받아 다음 단계의 작업을 수행한다.

---

# 🚀 Step 1. Trigger

## 역할

자동화를 시작하는 첫 번째 단계이다.

Google Docs에 새로운 회의록이 생성되면 Make가 이를 감지하여 워크플로우를 실행한다.

### 입력 데이터

* 문서 제목
* 문서 생성 시간
* 문서 ID

### 출력 데이터

* 문서 정보
* 본문을 가져오기 위한 참조 정보

---

## 분석

Trigger는 자동화의 시작점으로, 이벤트가 발생하지 않으면 이후 모든 작업도 실행되지 않는다.

즉, Trigger는 워크플로우 전체를 활성화하는 핵심 요소이다.

---

# 📄 Step 2. 문서 내용 추출

## 역할

Trigger가 전달한 문서 정보를 이용하여 실제 회의 내용을 읽어온다.

이 단계에서는 AI가 이해할 수 있도록 문서의 텍스트를 추출한다.

### 입력 데이터

* 문서 ID

### 처리 내용

* 제목 추출
* 본문 추출
* 텍스트 정리

### 출력 데이터

* 회의록 전체 내용

---

## 분석

회의록이 제대로 추출되지 않으면 이후 AI 요약도 정확하게 수행될 수 없다.

따라서 자동화 과정에서 가장 중요한 데이터 수집 단계라고 할 수 있다.

---

# 🤖 Step 3. AI 요약

## 역할

추출된 회의 내용을 ChatGPT에 전달하여 핵심 내용을 자동으로 생성한다.

이번 프로젝트에서는 다음 항목을 중심으로 요약하도록 설계하였다.

* 📌 회의 주제
* 📌 주요 논의 내용
* 📌 결정 사항
* 📌 담당 업무
* 📌 다음 일정

---

### 입력 데이터

* 회의록 원문

### 처리 내용

* 자연어 분석
* 핵심 내용 추출
* 일정한 형식으로 재구성

### 출력 데이터

* 요약된 회의 내용

---

## 분석

AI를 활용함으로써 사람이 직접 회의록을 읽고 요약하는 시간을 크게 줄일 수 있었다.

또한 일정한 형식으로 결과를 생성하기 때문에 회의록의 품질을 일정하게 유지할 수 있었다.

---

# 🔀 Step 4. Filter

## 역할

AI가 생성한 결과를 검사하여 다음 작업을 수행할지 결정한다.

이번 프로젝트에서는 **결정 사항(Action Items)** 이 존재하는 경우에만 다음 단계를 실행하도록 구성하였다.

---

### 입력 데이터

* AI 요약 결과

### 조건

```text
결정 사항 존재 여부
```

---

### 실행 결과

✔ 결정 사항 있음

↓

Discord 전송

---

❌ 결정 사항 없음

↓

Workflow 종료

---

## 분석

Filter를 사용하지 않는다면 모든 회의록이 Slack으로 전송되어 불필요한 알림이 증가할 수 있다.

조건 분기를 추가함으로써 필요한 경우에만 알림을 전송하도록 개선하였다.

---

# 💬 Step 5. Slack 메시지 전송

회의 요약을 팀 채널에 자동으로 공유한다.

Google Sheets를 함께 사용하면 회의 데이터를 체계적으로 관리할 수 있으며, 이후 다른 자동화나 데이터 분석에도 활용할 수 있다.

# 🔄 데이터 흐름 분석

이번 프로젝트에서는 하나의 데이터가 여러 단계를 거치며 점차 가공되는 구조를 확인할 수 있었다.

```text
회의록

↓

원본 데이터

↓

AI 분석

↓

요약 데이터

↓

Discord 공유

↓

Google Sheets 저장
```

데이터는 각 단계에서 새로운 정보를 추가하거나 가공하면서 최종 결과를 생성하였다.

이러한 구조는 ETL(Extract, Transform, Load) 방식과 유사한 데이터 처리 흐름을 가진다.

* **Extract** : Google Docs에서 데이터 추출
* **Transform** : ChatGPT를 이용한 요약 및 가공
* **Load** : Discord 공유 및 Google Sheets 저장

# 🤖 AI 연동 기능

## AI를 활용한 이유

기존의 자동화는 데이터를 단순히 전달하거나 저장하는 역할에 집중되어 있다. 하지만 최근에는 생성형 AI를 활용하여 데이터를 분석하고 가공하는 것이 가능해졌다.

이번 프로젝트에서는 **제미나이**를 활용하여 회의록의 핵심 내용을 자동으로 요약하도록 설계하였다.

이를 통해 사람이 직접 긴 회의록을 읽고 요약하는 시간을 줄이고, 일관된 형식의 결과를 생성할 수 있도록 하였다.

# ⚠️ 오류 처리 전략

## 오류 처리가 필요한 이유

자동화 워크플로우는 여러 외부 서비스를 연결하여 동작하기 때문에 다양한 원인으로 오류가 발생할 수 있다.

예를 들어,

* Google API 응답 실패
* Slack 서버 오류
* 인터넷 연결 문제
* AI 응답 실패
* 권한 오류

등이 발생할 가능성이 있다.

이러한 오류를 고려하지 않으면 자동화가 중간에서 종료되어 업무 누락으로 이어질 수 있다.

---

# 🚨 예상 가능한 오류

| 오류 상황               | 원인      | 영향           |
| :------------------ | :------ | :----------- |
| Google Docs 접근 실패   | 권한 오류   | 회의록을 읽을 수 없음 |
| ChatGPT 응답 실패       | API 오류  | 요약 생성 실패     |
| Slack 전송 실패         | 네트워크 문제 | 팀원에게 전달되지 않음 |
| Google Sheets 저장 실패 | API 제한  | 회의 이력 누락     |

---

# 🔔 실패 알림 설계

오류가 발생할 경우 관리자가 즉시 확인할 수 있도록 알림을 전송하도록 설계하였다.

예시

```text
⚠️ Automation Failed

Workflow : Meeting Summary

Failed Step : Slack Send Message

Time : 2026-07-08 15:30

Status : Retry Required
```

이와 같은 알림을 이메일 또는 메신저로 전달하면 문제를 빠르게 확인하고 대응할 수 있다.

---

# 🔄 재시도(Retry) 전략

일시적인 네트워크 문제나 API 오류는 일정 시간이 지난 후 정상적으로 처리되는 경우가 많다.

따라서 다음과 같은 재시도 전략을 적용할 수 있다.

```text
실패 발생

↓

30초 대기

↓

다시 실행

↓

성공

또는

3회 실패

↓

관리자 알림
```

이러한 방식은 불필요한 수작업을 줄이고 자동화의 안정성을 높여준다.

---

# 🗂️ 대체 경로(Fallback) 설계

특정 서비스가 실패하더라도 데이터를 완전히 잃지 않도록 대체 경로를 설계하였다.

예를 들어 Slack 전송에 실패한 경우에는 회의 요약을 **임시 Google Sheets(Error Log)** 에 저장하도록 구성할 수 있다.

```text
Slack 전송 실패

↓

Error Sheet 저장

↓

관리자 확인

↓

수동 전송
```

이와 같은 Fallback 전략은 데이터 손실을 최소화하는 데 효과적이다.

# 💡 프로젝트를 통해 배운 점

이번 프로젝트를 진행하면서 단순히 자동화 도구의 사용법을 익히는 것을 넘어, 자동화를 설계하는 사고방식의 중요성을 배울 수 있었다.

특히 다음과 같은 점을 새롭게 이해할 수 있었다.

### 🚀 1. 자동화는 업무를 분석하는 것에서 시작된다.

자동화 도구를 사용하는 것보다 먼저 **어떤 업무가 반복되고 있는지 파악하는 과정**이 중요했다.

반복 업무를 정확하게 분석해야 적절한 Trigger와 Action을 설계할 수 있으며, 불필요한 작업을 제거하여 더욱 효율적인 워크플로우를 만들 수 있었다.

---

### 🔄 2. 조건 분기는 자동화의 핵심 요소이다.

모든 데이터를 동일하게 처리하는 것이 아니라, 상황에 따라 다른 작업을 수행하도록 설계하는 것이 실제 업무 환경에서는 매우 중요했다.

Filter와 Router를 적절히 활용하면 불필요한 작업을 줄이고 필요한 작업만 수행할 수 있다는 점을 확인하였다.

---

### 🤖 3. 생성형 AI는 자동화의 활용 범위를 넓혀준다.

기존의 자동화는 데이터를 전달하거나 저장하는 역할이 중심이었다.

하지만 생성형 AI를 활용하면 데이터를 **분석하고 요약하며 새로운 정보를 생성**할 수 있기 때문에 자동화의 활용 범위가 더욱 넓어진다는 점을 경험하였다.

---

### 📊 4. 유지보수를 고려한 설계가 중요하다.

자동화는 한 번 구현하고 끝나는 것이 아니라 지속적으로 수정되고 확장될 가능성이 높다.

따라서 워크플로우를 명확하게 설계하고, 각 단계의 역할을 구분하며, 오류 처리 전략을 함께 고려하는 것이 중요하다는 점을 배울 수 있었다.
